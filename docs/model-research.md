# 前期调研

## 调研目标

在正式写代码前，先明确这个项目的技术路线是否适合当前学习目标，并筛选出最适合零预算起步的模型和整体方案。

## 项目形态调研

一开始有几个可选方向：

- 纯聊天机器人
- AI 工单助手
- AI 知识库问答系统
- 多智能体平台

最终选择 `AI 工单与知识库 Copilot`，原因如下：

- 比纯聊天机器人更接近真实业务场景
- 能覆盖聊天、工具调用、检索、引用、评测这些 AI 应用核心能力
- 比多智能体平台简单，适合从零到一完整做完
- 更容易在面试时讲清楚业务价值和技术实现

## 大模型平台调研

调研时重点看了这几件事：

- 是否适合零预算起步
- 是否支持基础聊天
- 是否支持流式输出
- 是否支持工具调用
- 是否容易在国内使用

### 1. 讯飞星火

优点：

- 国内可用
- 有免费模型

问题：

- 免费能力更适合基础聊天
- 对工具调用等 AI 应用核心能力的支持不适合作为首选主线

结论：

可以作为备选，不作为本项目首发方案。

### 2. 阿里百炼 / Qwen

优点：

- 能力完整
- 适合做更完整的 AI 应用

问题：

- 虽然有免费额度，但整体更适合“有一定额度预算”的场景

结论：

适合作为后续扩展或备选路线，不作为当前零预算主线。

### 3. Gemini

优点：

- 官方文档和生态都比较成熟

问题：

- 地区使用限制对当前场景不够友好

结论：

不作为当前主线方案。

### 4. DeepSeek

优点：

- 成本低
- 模型能力强

问题：

- 不属于严格意义上的零成本起步方案

结论：

可以作为以后愿意付少量费用时的备选。

### 5. GLM

优点：

- 更适合当前零预算起步
- 足够支持首版项目所需的主要能力
- 接入门槛低，适合作为学习项目主线

结论：

当前项目主模型确定为 `GLM-4.7-Flash`。

## 技术框架调研

### 为什么前端选 React 19

- 项目本身就是要做 AI 应用，而不是只做后端接口
- React 更适合和 AI 聊天、流式消息、状态管理这类前端交互结合
- 直接使用当前主流版本更有长期价值

### 为什么后端选 Node.js + Express 5

- 当前目标是先把 AI 应用主链路跑通
- Node 更适合快速完成前后端联调
- Express 生态更成熟，资料更多
- 单体服务更适合当前学习阶段

### 为什么不先用 Go

- Go 更适合后续对性能、异步任务或工程化做专项增强
- 但当前阶段最重要的不是极致性能，而是先把 AI 应用闭环做完整

## 当前整体结论

项目当前的总路线是：

- 产品形态：`AI 工单与知识库 Copilot`
- 前端：`React 19`
- 后端：`Node.js + Express 5`
- 数据库：`SQLite`
- 模型：`GLM-4.7-Flash`
- 服务形态：`模块化单体`
- MCP：`首版不需要`

## 后续可能的升级方向

- 检索从 `SQLite FTS` 升级到向量检索
- 工单系统从 mock 换成真实平台
- 引入更完整的评测和可观测性
- 在某些模块上再对比 Go 版本实现
